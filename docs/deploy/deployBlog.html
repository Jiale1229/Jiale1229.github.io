自动化部署项目

自动化部署项目

🙆‍♂️ 徐乐乐 🗓️ 2022/10/14 🏷️ GitHub Actions

前言

基于 vuepress 博客，利用CLI 本地构建结合 GitHub Action自动化部署到远程仓库上。最终实现：在本地push代码到远程仓库的时候，能够自动部署到 GitHub Pages。

# 建仓

提示

如果建立的仓库名称是 username.github.io ，则最后访问的地址 username.github.io
如果不是这个名称，则最后访问的地址是 username.github.io/repo/ ，repo 是仓库名字，此时还需要先修改.vuepress/config.js

// .vuepress/config.js
module.exports = {
    base: "/repo/"
}

# 分仓

有两种做法 两者只是存放源码和静态资源的仓库不同，部署的过程和方式都是相同的

# Case1 （推荐）

将源码和静态资源放在 两个仓库 来实现

优点

一个仓库用于日常的开发,新增文章或者二次开发，不断迭代，

另一个仓库只用于线上预览，项目中最终构建 build 生成的public目录文件推送到这个远程仓库当中，

保持两个仓库的相互独立,互不干扰,如果全部放在一个仓库里,不易于后续维护和代码的管理

# Case2

将源码和静态资源放在 同一个仓库不同的分支 来实现。

设置访问路径

打开仓库设置的 GitHub Pages 模块，

选择 存放静态资源的分支 为目标分支，

根目录为 root

# 本地CLI手动打包发布

注意

需要将 USERNAME 和 REPO 替换成自己的GitHub账号和仓库名称 这里我拿自己的GitHub账号为例。

# 1.CLI脚本配置

在项目根目录下，创建一个 deploy.sh 文件（自行判断去掉高亮行的注释）

# 确保脚本抛出遇到的错误
set -e


# 打包生成静态文件
npm run build

# 进入打包好的文件夹
cd public

git init
git add -A
git commit -m 'deploy'

# 如果是发布到自定义域名
# echo 'www.example.com' > CNAME

# 如果发布到 https://<USERNAME>.github.io
# 访问地址 https://jiale1229.github.io/
# git push -f git@github.com:Jiale1229.github.io.git master

# 如果发布到 https://<USERNAME>.github.io/<REPO>
# 访问地址 https://jiale1229.github.io/xulele/
# 将静态文件部署在当前仓库 gh-pages 分支
# git push -f git@github.com:Jiale1229/xulele.git master:gh-pages

cd -

# 2.添加执行命令

打开package.json下新增一条命令

"scripts": {
  ...,
  "dep": "bash deploy.sh"
}

# 3.运行命令

npm run dep & yarn dep

之后我们在完成文档编辑+页面维护之后，只需要运行一下命令即可完成 本地打包、上传到远程仓库发布，等待推送成功后，打开访问地址就可以看到页面啦。

# Github Actions 线上打包发布

实现

结合本地cli运行命令只需要push源码到远程仓库。剩下的交给Github Actions 线上 打包，发布，实现真正意义上的自动化部署！关于 Github Actions 了解更多 (opens new window)

# 1.修改deply.sh配置

push源码到远程仓库

# 确保脚本抛出遇到的错误
set -e

git init
git add -A
git commit -m 'deploy'

git push -f git@github.com:Jiale1229/xulele.git master:main

cd -

# 2.创建yml配置文件

在项目根目录下新建 .github/workflows/main.yml ，github会自动运行workflows目录下所有的yml文件。

# 3.内容编写解析

# 构建流程的名称
name: Deploy GitHub Pages

# 触发方式：在 push 到 main 分支后
on:
  push:
    branches:
      - main
# 任务
jobs:
  build-and-deploy:
    concurrency: ci-${{ github.ref }}
    # 任务运行的服务器环境：最新版 Ubuntu
    runs-on: ubuntu-latest
    # 该任务所包含的步骤
    steps:
      # 步骤名称 拉取当前库的代码
      - name: Checkout 🛎️
        # 步骤所依赖的操作库 这里引用了官方发布的git操作库 将对应仓库的源码克隆到当前工作流中
        uses: actions/checkout@v3

      # 生成静态文件
      - name: Install and Build 🔧
        run: yarn && yarn build

      # 部署到 GitHub Pages
      # 查看 workflow 的文档来获取更多信息
      # https://github.com/marketplace/actions/deploy-to-github-pages
      - name: Deploy GitHub Pages 🚀
        uses: JamesIves/github-pages-deploy-action@v4.4.1
        with:
          folder: public  # 静态文件的输出目录
          token: ${{ secrets.MY_PAT }} # 个人访问令牌 Personal access tokens
          repository-name: Jiale1229/Jiale1229.github.io  # 指定要部署的仓库
          branch: main # 希望部署到的分支 默认值为 gh-pages

      # 部署到自己的服务器
      # 查看 workflow 的文档来获取更多信息
      # https://github.com/marketplace/actions/ssh-deploy
      - name: Deploy MY Server 🌐
        uses: easingthemes/ssh-deploy@v2.2.11
        env:
          SSH_PRIVATE_KEY: ${{ secrets.SSH_PRIVATE_KEY }} # 服务器SSH私钥
          REMOTE_HOST: ${{ secrets.REMOTE_HOST }} # 服务器域名
          TARGET: ${{ secrets.TARGET }} # 服务器静态文件存放的目录
          REMOTE_PORT: ${{ secrets.REMOTE_PORT }}  # 端口
          REMOTE_USER: root # 一般默认用户名为root
          SOURCE: public # 静态文件的输出目录

# 上述名为"Deploy GitHub Pages"的工作流程：

  1. 当本地向远程仓库的main分支push最新代码时触发后续任务，

  2. 该工作流程中有一个名为 “build-and-deploy” 的任务，且该任务运行在 “ubuntu-latest” 虚拟环境下。通过依次执行其包含的多个步骤，可实现整个自动化部署功能。

    • 使用git操作库 actions/checkout@v3 (opens new window) 将对应仓库的源码克隆到当前工作流中

    • 安装项目依赖，并打包生成静态文件 public目录

    • 使用部署库 JamesIves/github-pages-deploy-action@v4.4.0 (opens new window) ，将 public 目录中的所有文件部署到 Jiale1229/Jiale1229.github.io 的 main 分支。需要配置 token 变量，如果未指定要部署的仓库，则默认部署到当前仓库的 gh-pages分支

    • 使用部署库 easingthemes/ssh-deploy@v2.2.11 (opens new window) 将项目部署到自己的服务器，前提需要配置好相应的变量

# 配置 token

  1. 创建PAT(Personal Access Token)

    • 创建PAT的主要目的是给当前库访问目标库的权限，可以让当前库的actions推送构建好的代码到目标库中去
    • Profile Photo -> Settings -> Developer settings -> Personal access tokens -> Generate new token
    • 勾选下面两个scope: workflow & write:packages
    • PAT生成好之后，复制下来

  2. 在当前库中设置Secret

    • 在私有库中: Settings -> Secrets -> Actions -> New repository secret
    • 粘贴刚才生成的PAT值粘贴 到value框中，并命名这个secret为MY_PAT

# 4.运行本地CLI命令，走你

npm run dep & yarn dep

推送完成之后，可以在仓库的 Actions 看到正在进行的工作流。

如果构建失败GitHub会发送一封邮件到你的邮箱。可以在对应工作流程运行的详情页中查看报错信息，根据报错信息定位问题所在并解决。
